# 产品路线图

## 文档说明

本文档用于沉淀当前版本的产品路线图，基于现有仓库能力、用户视角分析和产品经理视角的优先级整理而成。

当前产品已经不只是一个“简历编辑器”，而是一个围绕 **本地简历库 + AI 定制/优化 + 求职看板** 展开的桌面端求职工作流工具。后续规划的重点，不是零散地补功能点，而是把已经存在的能力串成更完整、更可追踪、更易执行的求职闭环。

## 产品定位判断

- **当前核心定位**：本地优先的桌面求职管理工具
- **当前能力基础**：多份简历管理、模块化编辑、AI JD 分析、AI 优化、导出、求职看板
- **下一阶段产品重点**：从“能写简历、能导出”升级到“能管理版本、能跟踪投递、能理解 AI 修改结果、能支持不同求职策略”

## 用户视角总结

### 1. 新人 / 校招生

这类用户的核心问题通常不是“没有编辑器”，而是不知道接下来应该做什么，也不知道自己的简历为什么匹配或不匹配岗位。

重点需求：

- 求职看板中的下一步行动提示
- JD 匹配结果的解释能力，而不只是一个分数
- 面试准备空间
- 更适合新人的起步模板或引导路径
- 关键词建议与真实经历之间的证据约束
- 能看懂不同简历版本之间发生了什么变化

### 2. 有经验的转岗 / 晋升型用户

这类用户往往会同时准备多个版本，例如 IC / 管理岗、不同业务方向、不同资历叙事角度。

重点需求：

- 简历变体家族管理
- 父子版本关系与来源追踪
- 同一申请关联多个候选简历版本
- 版本差异比较
- 按策略标签筛选不同版本
- 更完整的投递进度与阶段追踪

### 3. 高频投递用户

这类用户最在意的是效率，痛点集中在重复操作、频繁切换页面和单条处理流程过长。

重点需求：

- 看板批量操作
- 跟进动作、跟进时间、最后联系时间、来源、结果等结构化字段
- 更紧密的看板与简历库联动
- 批量 JD 分析 / 批量定制简历
- 导出预设与一键发送材料包
- 投递结果回顾与转化分析

### 4. 导师 / 教练 / 审阅者

这类用户未必需要云协作，但需要更清晰的本地审阅工作流。

重点需求：

- 分段评论与已处理状态
- 审阅 rubric / 质量门槛
- 更成体系的模板库
- 一键生成审阅包（预览、JSON、JD、备注）
- 基础版 / 优化版 / JD 定制版的对比与确认流程

## 优先级定义

- **P0**：明显影响主线闭环，应优先投入的能力
- **P1**：在主线闭环建立后，显著提升效率和适配性的能力
- **P2**：用于增强复盘、审阅和长期成长体验的能力

## 优先级路线图

### P0

#### 1. 简历版本管理中心

目标：让 clone、AI 优化、JD 定制后的多个副本不再是松散文件，而是可追踪、可比较、可回溯的版本体系。

建议范围：

- 变体家族（variant family）
- 父子版本关系
- 标记某个投递实际使用了哪个版本
- 不同版本之间的差异比较
- 清楚区分基础版、优化版、JD 定制版

为什么优先：

- 现有产品已经具备生成多个副本的能力
- 但用户无法直观看到这些副本之间的关系
- 这会直接影响 AI 结果可理解性、投递可追踪性和后续复用效率

#### 2. 求职看板动作化

目标：让看板不只是“状态栏”，而是可驱动下一步行动的执行面板。

建议范围：

- 下一步动作（next action）
- 跟进日期（follow-up date）
- 面试阶段 / 轮次
- 提醒状态
- 审阅状态或阻塞状态
- 最后联系时间、来源、结果等结构化字段

为什么优先：

- 当前看板已经是首页主入口
- 但仍偏向静态记录，缺少后续动作管理
- 对新手、高频投递者和复杂流程用户，这都是高频刚需

#### 3. AI 修改解释与前后对比

目标：让用户知道 AI 改了什么、为什么改、哪些建议已经采用、哪些尚未采用。

建议范围：

- 优化前 / 优化后对比
- JD 定制前 / 定制后对比
- 建议原因解释
- 已应用 / 未应用建议状态
- 关键关键词覆盖变化说明

为什么优先：

- 当前 AI 能生成 patch 或新副本
- 但用户难以理解变化质量，尤其是新手和谨慎型用户
- 这会降低 AI 功能的信任度和复用率

### P1

#### 4. 人群 / 岗位导向模板系统

目标：从“默认模板 + 示例模板”升级为更产品化的模板路径。

建议范围：

- 校招生模板
- 社招模板
- 管理岗 / 专业岗模板
- 不同岗位叙事策略模板
- 使用场景导向的起步入口

#### 5. 批量工作流

目标：降低高频重复操作带来的摩擦成本。

建议范围：

- 批量更新投递状态
- 批量 JD 分析
- 批量生成定制简历
- 导出预设
- 一键导出材料包

#### 6. 面试准备工作区

目标：把“投递之后”这段流程也纳入产品主线，而不是停留在简历导出结束。

建议范围：

- 针对岗位的面试要点整理
- JD 风险点与补充准备项
- 面试问题准备区
- 申请记录与面试准备内容联动

### P2

#### 7. 求职复盘与数据分析

目标：帮助用户理解哪些策略更有效，而不是只看到零散的投递记录。

建议范围：

- 投递转化漏斗
- 不同简历版本的效果回顾
- 关键词覆盖与投递结果的关联观察
- 按岗位方向 / 策略标签做复盘

#### 8. 导师 / 审阅工作流

目标：为本地优先场景下的教练式审阅提供更完整的支撑。

建议范围：

- 分段评论
- 评论处理状态
- 审阅 rubric
- 审阅包导出
- 版本确认 / 批注式比较

## 分阶段路线图

### Phase 1（1-2 个版本）

聚焦建立主线闭环：

- 简历变体家族与版本关系
- 看板中的下一步动作、跟进时间、面试阶段
- AI 优化 / JD 定制前后对比
- 更强的“投递记录 ↔ 简历版本”关联

这阶段的目标是让用户能回答四个关键问题：

1. 我现在投到了哪一步？
2. 我这次投递到底用了哪个版本？
3. AI 改了什么？
4. 我下一步应该做什么？

### Phase 2（2-3 个版本）

聚焦效率与适配性提升：

- 批量工作流
- 导出预设
- 人群 / 岗位模板系统
- 面试准备工作区

这阶段的目标是减少重复劳动，并让不同类型用户能更快进入适合自己的工作流。

### Phase 3（后续演进）

聚焦长期成长与专业化使用：

- 求职复盘分析
- 导师 / 审阅工作流
- 更系统化的模板与策略库

这阶段的目标是让产品从“求职工具”进一步演进为“求职策略工作台”。

## 建议的推进顺序

如果只能先做一件事，优先做：

1. **简历版本管理中心**

因为它直接连接：

- AI 优化结果的可理解性
- JD 定制副本的可管理性
- 投递记录的可追踪性
- 后续比较、复盘、审阅等能力的基础数据结构

如果要同时启动第二条主线，则建议并行推进：

2. **求职看板动作化**

因为首页已经是看板，给它补足动作层，会最快拉高产品整体完成度和日常使用频率。

## 范围边界

当前路线图默认遵循以下边界：

- 保持 **桌面本地优先** 的产品方向
- 不把云协作、账号体系、在线多人协同作为短期主线
- 先补足求职主链路，再扩展更多外围能力

## 后续维护建议

- 每完成一个 Phase 的核心能力后，回看一次本文件
- 新增 roadmap 项目时，同时标注其属于 P0 / P1 / P2 中的哪一层
- 当产品首页、核心路由或主流程发生变化时，重新校准本路线图
